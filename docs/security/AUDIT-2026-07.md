# Internal Security Audit — July 2026

**Scope:** full source tree at `main` (35e2138), the four GitHub Actions
workflows, and the published PyPI artifact.
**Method:** manual review of the audit subsystem, key management, process
lifecycle, native backend, KV store, context memory, and CLI; Bandit SAST
(clean — three low-severity informational hits: guarded `try/except/pass`
in health polling and the argv-based `subprocess` usage); dependency and
workflow review.
**Remediation:** PR #47 (version 0.4.1, not released).

## Summary

The security core is sound. The review found **no injection, unsafe
deserialization (`pickle`/`eval`/`shell=True`), or plaintext-fallback
paths**; SQL is parameterized throughout; keys come from
`secrets.token_bytes`; the audit chain's canonical serialization is
length-prefixed (field boundaries cannot be forged, `NULL` ≠ `""`);
`verify()` is strictly read-only and never re-anchors; the constructor
fails closed when SQLCipher is absent. The documented "honest boundary"
(what the anchor does and does not guarantee) is accurate.

The significant findings cluster around the *operational correctness of
the tamper-evidence mechanism itself* — cases where the mechanism would
cry wolf (H1, M2) or go silently blind (M1), which is the failure mode
that matters most for a log whose value is being believed.

## Findings

| ID | Severity | Finding | Status |
|----|----------|---------|--------|
| H1 | High | Head anchor was machine-global, not per-database: two logs on one host clobbered each other's anchor, making an honest log verify as "replaced" and burying real alarms in false ones. | Fixed (PR #47): `anchor_scope(db_path)`, per-db keychain entries. |
| H2 | High (forward-looking) | `state_set` feeds raw bytes into `llama_state_seq_set_data` — C-side deserialization. Safe while blobs are in-memory and self-produced; becomes an untrusted-input-into-C-parser path the moment a disk-backed `KVStore` (already announced in its docstring) or blob sharing ships. | Open — required before any disk-backed store: HMAC-SHA256 (keychain-derived key) over persisted blobs, verified before `state_set`. Fuzzing note below. |
| M1 | Medium | `record()` ignored `store_head_anchor`'s return value: a keychain outage silently dropped the wholesale-replacement guarantee mid-run. | Fixed (PR #47): `anchor_failures` counter + one-time warning; anchor retried on next write. |
| M2 | Medium | Crash between row commit and anchor write left a stale anchor; `verify()` mis-diagnosed it as "history replaced". | Fixed (PR #47): `verify()` distinguishes an unanchored tail (`anchor_lag=N`) from a replacement/rollback (anchor names no row in the chain). Both remain `ok=False`. |
| M3 | Medium | The managed `llama-server` child runs without `--api-key`: any same-host process can reach the HTTP API (foreign prompts into slots, DoS; llama-server's built-in tools raise the ceiling). The `find_free_port` race additionally permits local impersonation of the server. | **Deferred by decision.** Level 3 is planned to split into a separate distribution, which changes the HTTP exposure model; revisited then. Until that split, the L2 adapter should be treated as single-user-host only. |
| M4 | Medium | `error_message = str(e)` flowed unbounded into the metadata-only log; provider exception text can embed URLs with tokens or payload fragments. | Fixed (PR #47): clipped to 200 chars before hashing, in the decorator and in `record()` itself. |
| L1 | Low | llama-server stdout/stderr on unread `PIPE`s: a child logging > 64 KiB blocks on write and hangs. | Fixed (PR #47): stderr to a temp file (tail included in startup-failure errors), stdout discarded. |
| L2 | Low | Actions pinned by mutable tags, including `pypa/gh-action-pypi-publish@release/v1` (a moving branch) on the publish path. | Fixed (PR #47): all workflows pinned to commit SHAs. |
| L3 | Low | `_singleton_lock` declared but unused; TOCTOU in `load_or_create_key` first run; no `0600` on the DB file. | Fixed (PR #47). |
| L4 | Low | The published PyPI artifact predates the OIDC release workflow and carries no attestations/provenance. | Closes automatically with the next release through `release.yml` (Trusted Publishing). |

## Explicitly reviewed and found sound

- **Chain design:** length-prefixed canonical hashing; `GENESIS` constant
  (empty-vs-never-existed distinguishable); row `id` deliberately outside
  the hash (order is enforced by the chain, ids are presentational).
- **`verify()` discipline:** read-only, never moves the anchor —
  inspection cannot bless a forged history. Covered by tests that tamper
  through raw `sqlite3`, as an attacker would.
- **Rollback detection:** restoring an older DB snapshot is caught (the
  newer anchor names no row in the old chain).
- **Fail-closed posture:** missing SQLCipher raises rather than writing
  plaintext; wrong key fails at open via a forced sanity read.
- **PRAGMA key interpolation** is not an injection surface: the hex form
  is produced from `secrets.token_bytes`/caller bytes.
- **Workflows:** least-privilege `permissions`, OIDC-scoped publish job,
  Scorecard + Bandit + CodeQL SARIF in place.

## Recommendations beyond PR #47

1. **Before a disk-backed `KVStore` (H2):** MAC persisted blobs; treat
   `state_set` as a trust boundary. This also reframes the fuzzing plan —
   Atheris over the pure-Python surfaces (`_canonical`, `content_key`)
   has little to find; the boundary worth fuzzing is llama.cpp's state
   parsing fed by `state_set`, or at minimum strict size/version header
   validation on our side before the bytes cross into C.
2. **When the L3 split lands (M3):** per-launch random `--api-key` for
   any managed HTTP server that remains, and verify the health-checked
   process owns the expected port before sending prompts.
3. **`docs/THREAT_MODEL.md`:** the threat model currently lives in
   module docstrings (`audit/log.py`, `audit/key_manager.py`) and
   `SECURITY.md`. Extracting it into one document — assets, attacker
   capabilities (with/without keychain write), guarantees in force per
   configuration (`head_anchored`, `allow_unencrypted`, `anchor_every`)
   — is low-effort and is exactly the artifact an EU AI Act Art. 12 /
   26(6) assurance case and a CRA-readiness narrative build on.
4. **Anchor migration note for operators:** after upgrading to 0.4.1, an
   existing log verifies with `head_anchored=False` until its first
   post-upgrade write re-anchors it under the scoped keychain entry. The
   legacy unscoped entry can be removed manually.
